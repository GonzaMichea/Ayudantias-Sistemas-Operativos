1 — Introducción a los Procesos

¿Qué es un proceso?

Un proceso es una instancia activa de un programa en ejecución. A diferencia de un programa (que es código estático en disco), el proceso incluye el estado de ejecución, los datos en memoria y todos los recursos del sistema operativo asignados a él.

Definición formal

Un proceso UNIX/Linux contiene: (1) un conjunto de estructuras de datos para su gestión, (2) 3 segmentos de memoria virtual y (3) posibles segmentos compartidos con otros procesos.

Estructuras de datos

Zona U, estructura task/proc — usadas por el kernel para gestionar el proceso.

3 Segmentos de memoria

  • • Texto — código ejecutable
  • • Datos — variables estáticas/globales
  • • Pila — stack de ejecución

Segmentos compartidos

El segmento de código puede ser compartido entre procesos que ejecutan el mismo programa.

Programa (estático)

  • Código binario almacenado en disco
  • No tiene estado de ejecución
  • Pasivo — no consume CPU ni RAM por sí solo
  • Puede dar origen a múltiples procesos

Proceso (dinámico)

  • Instancia activa del programa cargado en RAM
  • Tiene estado: registros, PC, datos en memoria
  • Consume CPU, memoria, archivos abiertos, etc.
  • Varios procesos pueden ejecutar el mismo programa

2 — Estructura del Proceso

2 modos de ejecución y 2 zonas de memoria

Todo proceso en UNIX/Linux puede ejecutarse en dos modos distintos y su espacio de memoria virtual se divide en dos zonas claramente separadas:

Modo Usuario

El proceso ejecuta código de la aplicación. Acceso restringido al hardware — no puede ejecutar instrucciones privilegiadas ni acceder directamente a periféricos.

Ej. el código de tu programa C corriendo normalmente.

Modo Sistema (Kernel)

El proceso ejecuta código del kernel tras una syscall o interrupción. Acceso total al hardware. El proceso no cambia — solo cambia el modo de privilegio del procesador.

Ej. el mismo proceso cuando llama a read(), write(), fork()…

El kernel es parte de la zona de memoria virtual del proceso — no es un espacio separado.

En UNIX/Linux el kernel está mapeado en la zona de sistema del espacio virtual de cada proceso. Esto permite que, cuando ocurre una syscall o interrupción, el procesador pueda cambiar de modo y empezar a ejecutar código del kernel sin cambiar el contexto del espacio de direcciones (sin cambiar el CR3 en x86).

Ventajas de este diseño:

  • La transición modo-usuario → modo-sistema es muy rápida (solo cambia el nivel de privilegio del procesador).
  • El kernel puede acceder directamente a las estructuras del proceso (Zona U) sin necesidad de un cambio de espacio de direcciones.

3 — Zona U y Estructura Task

El Contexto del proceso y la Zona U

Contexto del proceso

Espacio usuario

Informaciones de control

Zona U y estructura proc — metadatos del proceso mantenidos por el kernel.

PATH, HOME, USER, etc. Heredadas del proceso padre.

Conmutación de proceso (context switch)

Cuando el kernel decide cambiar de proceso: (1) guarda el Contexto Hardware del proceso saliente en su Zona U. (2) Carga el Contexto Hardware del proceso entrante desde su Zona U a los registros físicos del procesador. El proceso "reanuda" exactamente donde quedó.

Contenido completo de la Zona U

La Zona U es parte del espacio del proceso y puede ser swapped out a disco.

  • Contexto Hardware del proceso
  • UID y GID reales
  • Argumentos del proceso
  • Código de error del último system call
  • Informaciones sobre señales pendientes
  • Header del programa (para el linker)
  • Lista de archivos abiertos (file descriptors)
  • Puntero al vnode del directorio corriente
  • Estadísticas: CPU usado, límites, quotas
  • System Stack (pila del kernel para este proceso)

Stack Usuario (Pila de Ejecución)

  • Vive en la Zona Usuario
  • Usada cuando el proceso ejecuta en modo usuario
  • Contiene: variables locales, parámetros de funciones, dirección de retorno
  • Manipulada por el código de la aplicación

Pila Sistema (System Stack)

  • Vive dentro de la Zona U (zona sistema)
  • Usada cuando el proceso ejecuta en modo sistema (kernel)
  • Contiene: contexto de llamadas al kernel, parámetros de syscalls
  • Solo accesible y manipulable por el kernel
Cada proceso tiene dos pilas: una para modo usuario y otra para modo sistema. Al entrar al kernel (por syscall o interrupción), el procesador cambia automáticamente al Stack Sistema del proceso.

El kernel distingue dos niveles de metadatos del proceso según su necesidad de estar siempre en RAM:

Estructura proc / task — siempre en RAM

Contiene información mínima que el kernel necesita incluso cuando el proceso está durmiendo: PID, estado, prioridad, puntero a la Zona U. El scheduler necesita acceder a estos datos sin hacer swap-in.

Zona U — puede salir a disco

Contiene información que solo se necesita cuando el proceso está activamente ejecutando (contexto HW, archivos abiertos, etc.). Si el proceso está completamente suspendido (swapped out), su Zona U puede ir a disco junto con sus segmentos de datos y código.

4 — Segmentos de Memoria Compartidos

Compartición del segmento de código

Cuando múltiples procesos ejecutan el mismo programa, el sistema puede hacer que sus Text Segments apunten al mismo bloque físico de memoria. Esto ahorra RAM significativamente.

Campo xcount

La Estructura Text lleva un contador xcount que indica cuántos procesos comparten ese código en este momento. Cuando xcount llega a 0, el sistema puede liberar ese bloque de memoria física.

Se puede compartir

  • Text Segment — el código es de solo lectura, es seguro compartirlo

No se comparte (privado)

  • Data Segment — variables propias de cada proceso
  • Stack Segment — pila única de cada proceso
  • Zona U — contexto hardware individual

La respuesta está en la naturaleza de cada segmento:

Código — inmutable (read-only)

Las instrucciones del programa no cambian durante la ejecución. Si proceso 1 y proceso 2 ejecutan el mismo programa, la instrucción en la dirección X siempre es la misma para ambos. Compartirla es 100% seguro y eficiente.

Datos — mutable (read-write)

Las variables globales son propias de cada proceso. Si P1 y P2 ejecutan el mismo programa, cada uno tiene su propio contador, su propia variable x, etc. Si compartieran datos, una escritura de P1 afectaría a P2 — eso sería incorrecto.

Excepción: los segmentos de datos sí se comparten temporalmente justo después de un fork(), pero protegidos con el mecanismo Copy On Write. En cuanto alguno escribe, se genera una copia privada.

5 — Estados de un Proceso

Grafo de estados de un proceso UNIX/Linux

READY

El proceso está listo y en espera de que el scheduler le asigne la CPU. Puede haber muchos procesos READY simultáneamente.

Entrada: fork(), wakeup(), task_switch()

RUNNING USER MODE

El proceso tiene la CPU y ejecuta código de la aplicación (modo usuario).

Entrada: INTR return, Syscall return

RUNNING SYSTEM MODE

El proceso tiene la CPU ejecutando código del kernel (por syscall o interrupción).

Entrada: Interruption, System Call

SLEEPING

El proceso está bloqueado, esperando un evento externo (E/S terminada, señal, recurso disponible).

Entrada: sleep() — Salida: wakeup()

ZOMBIE

El proceso terminó con exit(). Espacio de trabajo liberado, pero la task persiste hasta que el padre haga wait().

Entrada: exit() — Salida: wait()

TERMINATED

El padre llamó a wait(). Todas las estructuras del proceso (Zona U, proc, task…) son liberadas definitivamente.

Entrada: wait()

Transiciones principales:

Parent Process ──fork()──▶ READY

READY ──task_switch()──▶ RUNNING SYSTEM MODE

RUNNING SYSTEM ──INTR/Syscall Return──▶ RUNNING USER MODE

RUNNING USER ──Interruption/Syscall──▶ RUNNING SYSTEM MODE

RUNNING SYSTEM ──sleep()──▶ SLEEPING ──wakeup()──▶ READY

RUNNING SYSTEM ──exit()──▶ ZOMBIE ──wait()──▶ TERMINATED

USER → SYSTEM (dos causas)

  • System Call: el proceso invoca explícitamente al kernel (read, write, fork, open, exit…). El hardware cambia el nivel de privilegio y el SP al System Stack.
  • Interrupción hardware: el procesador recibe una interrupción externa (timer, dispositivo de E/S, etc.) mientras ejecuta el proceso. El kernel toma control.

SYSTEM → USER (dos causas)

  • INTR Return: el kernel terminó de manejar la interrupción y retorna al código de usuario donde fue interrumpido.
  • Syscall Return: el kernel completó la syscall y retorna el resultado al código de usuario.

Cuando el proceso hace exit(), el kernel libera lo que ya no sirve pero preserva lo que el padre aún necesita leer:

Se libera al hacer exit()

  • Segmentos de memoria virtual (Text, Data, Stack)
  • Zonas en el área de Swap
  • Archivos abiertos (se cierran)
  • La mayor parte de la Zona U

Persiste hasta wait()

  • Estructura Task / proc — para que el padre pueda recuperar el PID y el código de salida (exit status)
  • El PID (reservado hasta que el padre lo recoja)
Si el padre nunca llama a wait(), el zombie persiste indefinidamente, ocupando una entrada en la tabla de procesos. Con suficientes zombies se puede agotar el límite de procesos del sistema.

6 — Creación de Procesos: fork()

Jerarquía PADRE ⇌ HIJO e Init

UNIX/Linux organiza todos los procesos del sistema en un árbol jerárquico. Cada proceso tiene exactamente un padre (salvo el primero). Para crear un proceso hijo se usa la syscall fork().

Proceso Init (PID 1)

El primer proceso, creado directamente por el Kernel tras el boot. Es la raíz del árbol de procesos. Todos los demás procesos son descendientes de Init.

Algoritmo de fork() — paso a paso

  1. 1

    Crear nuevo proceso y asignar un nuevo PID único.

  2. 2

    Inicializar la estructura task del hijo.

  3. 3

    Asignar zona de memoria virtual (espacio de trabajo) al hijo.

  4. 4

    Copiar la Zona U del padre al hijo.

  5. 5

    Compartir Text y Data Segments entre padre e hijo (mismo bloque físico).

  1. 6

    Copiar el Stack Segment del padre al hijo.

  2. 7

    Copiar el contexto hardware del padre al hijo (PC, registros, PSW, FPU).

  3. 8

    Poner el hijo en estado READY e insertarlo en la lista de ejecución.

  4. 9

    Activar bit COW=1 en todas las páginas del Data Segment de ambos procesos.

  5. 10

    Return 0 al hijo, Return PID_hijo al padre.

Copy On Write (COW)

El mecanismo Copy On Write evita copiar el segmento de datos en el momento de fork(). Las páginas se marcan como "protegidas" y solo se copian si alguno de los dos procesos las modifica:

①

Después del fork()

Padre e hijo apuntan al mismo Data Segment. El sistema pone el bit COW = 1 en todas sus páginas en las tablas de páginas de ambos procesos, marcándolas como solo-lectura.

②

Cuando un proceso escribe en una página COW

El hardware genera una excepción de protección. El kernel detecta que es una página COW, crea una copia privada de esa página para el proceso que escribe, y actualiza su tabla de páginas. Ahora ese proceso tiene su propia copia.

③

Resultado

Solo se copian las páginas que realmente se modifican. Si el hijo hace exec() inmediatamente, ninguna página de datos llega a copiarse → fork() es casi gratuito en práctica.

Tabla de Páginas tras fork() — bit COW activo en segmento de datos:

Uso de fork() en C

void CreateProcess() { int pid; pid = fork(); // Llamada al sistema — crea el proceso hijo switch (pid) { case -1: // ERROR — no se pudo crear el proceso perror("Error fork"); exit(1); break; case 0: // HIJO — fork() retorna 0 al proceso hijo RunChild(); break; default: // PADRE — fork() retorna PID del hijo al padre RunFather(); break; } }

El hijo comienza a ejecutarse inmediatamente después de la llamada a fork(), es decir, en el mismo punto del código donde el padre estaba cuando llamó a fork().

¿Por qué?

El paso 7 del algoritmo de fork() copia el contexto hardware del padre al hijo, incluyendo el registro PC (Program Counter). El PC del padre apunta a la instrucción siguiente a fork() — y ese mismo valor se copia al hijo. Por tanto, cuando el hijo se ejecuta, retoma exactamente desde ese punto.

La única diferencia es el valor de retorno de fork(): el padre ve el PID del hijo (≠ 0) y el hijo ve 0. Eso es lo que permite distinguirlos en el switch/case.

Sin COW (copiado inmediato)

Cada fork() copiaría completamente todos los segmentos de datos del padre al hijo. Si el padre tiene 500 MB de datos, el fork() tardaría un tiempo notable y consumiría 500 MB adicionales de RAM, incluso si el hijo va a hacer exec() inmediatamente y nunca usará esos datos.

Con COW (copia diferida)

fork() es casi instantáneo: solo se crean las tablas de páginas del hijo. La copia real de cada página ocurre solo si alguien la modifica. En el patrón fork()+exec(), el hijo nunca modifica nada antes de exec() → cero copias, fork() cuesta casi 0 memoria extra.

Patrón fork()+exec(): El shell hace fork() → el hijo llama inmediatamente exec() para correr el programa pedido → exec() reemplaza todos los segmentos → las páginas COW nunca se copian porque el hijo las descarta antes de tocarlas.

7 — Ejecución de otro programa: exec()

¿Qué hace exec()?

exec() reemplaza la imagen del proceso (código, datos y stack) por un nuevo programa ejecutable, sin crear un nuevo proceso. El proceso conserva su PID e identidad, pero su contenido cambia completamente.

Permanece igual

  • PID

    El proceso conserva su identificador — para el sistema es el mismo proceso.

  • FDs

    Los archivos abiertos (file descriptors) se heredan en el nuevo programa, salvo los marcados con O_CLOEXEC.

  • Zona U

    La Zona U se actualiza pero el proceso mantiene el mismo PID y archivos.

Cambia completamente

  • Text

    Nuevo segmento de código cargado desde disco. Si ya existe en RAM (otro proceso lo usa), el sistema lo comparte.

  • Data

    Nuevas variables globales y estáticas del nuevo programa (cargadas desde disco).

  • Stack

    Reinicializado (INIT system stack). La pila comienza desde cero para el nuevo programa.

  • PC

    El PC apunta al entry point del nuevo ejecutable.

⚠ Valor de retorno especial de exec()

exec() no retorna si tiene éxito. El proceso original ya no existe — fue reemplazado por el nuevo programa. Solo retorna (con valor -1 y errno) si ocurre un error (ej. archivo no encontrado, sin permisos). Esto se debe a que el código que llamó exec() ya no existe en memoria.

Diagrama fork() + exec() — patrón del shell

SHELLRecibe comando del usuario
↓llama a fork()

PADRE (Shell)

Llama a wait() y espera a que el hijo termine. Cuando termina, muestra el prompt nuevamente.

HIJO

Llama a exec() con el comando pedido (ej. /bin/ls). La imagen del hijo se reemplaza por ls. Ejecuta ls, termina con exit().

El razonamiento es simple: exec() reemplaza el segmento de código (Text Segment) del proceso por el del nuevo programa. Esto significa que:

  1. El código original (que contenía la llamada a exec()) ya no existe en memoria.
  2. No hay ningún punto del código original al cual retornar.
  3. El PC del proceso fue cambiado al entry point del nuevo ejecutable.
  4. La ejecución continúa en el nuevo programa, no después de exec().
Si exec() falla (archivo no existe, sin permisos), el segmento de código original no fue reemplazado. El proceso sigue intacto y exec() puede retornar el error al código que lo llamó.

8 — Terminación de Procesos

Ciclo completo de terminación

Cuando un proceso llama a exit(), la terminación no es inmediata: el proceso pasa por el estado ZOMBIE antes de ser definitivamente eliminado.

1

El proceso llama a exit()

El sistema libera el espacio de trabajo: memoria virtual, zonas en swap, archivos abiertos. Pero NO la estructura task. El sistema elige otro proceso READY para ejecutar.

Z

Estado ZOMBIE

El proceso queda "muerto pero no enterrado". La estructura task persiste con el PID y el exit status (código de salida). El sistema envía la señal SIGCHLD al proceso padre para notificarle.

3

El padre llama a wait()

wait() retorna al padre el PID del hijo terminado y su código de salida. A partir de este momento el proceso puede pasar a TERMINATED.

4

Estado TERMINATED — liberación total

El sistema libera definitivamente todas las estructuras restantes: Zona U, proc, task, PID. El proceso ya no existe en el sistema.

Señal SIGCHLD

Cuando un hijo hace exit(), el kernel envía SIGCHLD al padre. El padre puede:

  • Ignorarla (default) — el zombie persiste hasta que el padre haga wait()
  • Instalar un handler que llame a wait() de forma asíncrona
  • Llamar a wait() explícitamente (blocking o non-blocking)

Procesos huérfanos y el rol de Init

Si el padre termina antes que el hijo, el hijo se convierte en huérfano. El kernel reasigna Init (PID 1) como nuevo padre. Init está permanentemente en wait(), por lo que recogerá al huérfano cuando este termine → evita zombies permanentes.

Acumulación de procesos ZOMBIE

Cada hijo que termina sin que el padre haga wait() deja una entrada en la tabla de procesos del kernel (la estructura task). Aunque no consume CPU ni memoria de usuario, sí ocupa una entrada de PID en la tabla.

Agotamiento de la tabla de procesos

El kernel tiene un límite máximo de procesos simultáneos (configurable). Si una cantidad suficiente de zombies acumula PIDs sin liberarlos, el sistema puede llegar a ese límite y rechazar la creación de nuevos procesos — incluso para root.

Buena práctica: Siempre llamar a wait() o waitpid() por cada fork(), o instalar un handler de SIGCHLD que lo haga. Alternativamente, hacer doble fork() (el nieto queda huérfano y es adoptado por Init).
  1. El shell (bash) está en estado RUNNING USER MODE esperando entrada del usuario.
  2. El usuario escribe ls -l. El shell interpreta el comando y llama a fork() → entra en RUNNING SYSTEM MODE.
  3. El kernel ejecuta el algoritmo de fork(): crea el proceso hijo, le asigna PID, copia Zona U, comparte segmentos de texto/datos con COW, copia Stack, pone el hijo en READY.
  4. El hijo eventualmente llega a RUNNING SYSTEM MODE, sale a RUNNING USER MODE. El hijo llama a exec("/bin/ls", ["-l"]).
  5. exec() reemplaza la imagen del hijo: carga el código de /bin/ls, inicializa nuevos datos y stack, actualiza el PC. El hijo ahora es ls.
  6. ls ejecuta, imprime el listado. El padre (bash) está en estado SLEEPING — llamó a wait().
  7. ls termina con exit(0). El proceso pasa a estado ZOMBIE. El kernel envía SIGCHLD al bash.
  8. Bash despierta del wait() → recibe el PID de ls y su exit status (0 = éxito).
  9. El proceso ls pasa a estado TERMINATED. Sus estructuras se liberan.
  10. Bash vuelve a RUNNING USER MODE y muestra el prompt.

Estructura task/proc (siempre en RAM)

  • Prioridad de scheduling
  • Puntero a la Zona U
  • Punteros a segmentos de memoria

Solo necesaria cuando el proceso está activamente ejecutando.

  • System Stack
  • Lista de archivos abiertos

Tras fork()

El hijo hereda todos los file descriptors del padre. Ambos procesos comparten los mismos archivos abiertos (mismas entradas en la tabla de archivos del kernel, mismo offset de lectura/escritura).

Cuidado: si padre e hijo escriben al mismo archivo sin coordinación, puede haber corrupción de datos.

Tras exec()

Los file descriptors se conservan por defecto en el nuevo programa (el FD 0/1/2 para stdin/stdout/stderr se mantiene). Esto permite redireccionamiento de E/S: el shell configura los FDs antes del exec() del hijo.

Los FDs marcados con O_CLOEXEC o FD_CLOEXEC se cierran automáticamente al hacer exec().

Compartido realmente

Text Segment (código) — es inmutable, compartirlo es seguro y eficiente permanentemente.

Compartido con COW

Data Segment — compartido inicialmente, se copia página a página solo al escribir.

Privado (copiado)

Stack Segment y Zona U — se copian en el fork(); nunca se comparten.

Garantía de independencia del COW: desde el punto de vista del programador, padre e hijo tienen espacios de datos completamente independientes. El COW es un detalle de implementación transparente — solo cambia cuándo ocurre la copia, no si ocurre. El primer write a una página compartida siempre genera la copia, garantizando que una modificación de un proceso nunca afecte al otro.
