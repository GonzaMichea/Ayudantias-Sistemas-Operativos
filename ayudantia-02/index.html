Guía de Estudio: Scheduling y Gestión del Tiempo

Parte 1: Resumen Teórico

1. Conceptos Fundamentales

Programa

Expresión estática de un algoritmo en un lenguaje de programación. Es simplemente código pasivo almacenado en disco.

Tarea

La unión de un programa con los datos iniciales a los cuales se va a aplicar. Aún no se está ejecutando.

Proceso

La ejecución viva de una tarea. Posee un contexto real (estado de registros de CPU, Program Counter, memoria RAM asignada).

El rol del Sistema Operativo en el Scheduling

El SO actúa como árbitro de los recursos (CPU, Memoria, E/S). Su objetivo principal al planificar (Scheduling) es encontrar un equilibrio entre:

  • Satisfacer al usuario: Tiempos de espera y de respuesta cortos.
  • Eficiencia del sistema (Throughput): Maximizar el número de tareas terminadas por segundo y minimizar el tiempo ocioso (Idle) del procesador.

2. Algoritmos de Scheduling (Planificación)

El Modelo de Planificación dicta la filosofía (Ej: Batch Processing vs Tiempo Compartido), mientras que el Algoritmo de Scheduling es el mecanismo matemático/lógico que decide quién entra a la CPU cuando esta se libera.

Sin Requisición (Non-Preemptive)

Modelo Batch. La tarea toma la CPU y no la suelta hasta que termina o hace E/S.

  • FCFS (First Come First Served): Orden de llegada. Muy fácil de programar, pero las tareas largas bloquean a las cortas causando frustración.
  • SJN (Shortest Job Next): Prioriza la tarea con menor tiempo total de ejecución. Minimiza drásticamente el tiempo de espera promedio, pero sufre de inanición (starvation): si siempre llegan tareas cortas, una larga jamás se ejecutará.
  • HPRN (Highest Penalty Ratio Next): Mejora de SJN. Calcula una "tasa de penalización" P = (Espera + Duración) / Duración. Así, las tareas largas ganan prioridad a medida que envejecen esperando, evitando la inanición.

Con Requisición (Preemptive)

La CPU puede ser arrebatada a la fuerza si llega una tarea más prioritaria.

  • PSJN (Preemptive SJN): Si llega una tarea nueva y su duración total es menor que la tarea actual, la interrumpe. Problema: puede interrumpir tareas que ya estaban casi terminadas.
  • SRTF (Shortest Remaining Time First): Mucho más inteligente que PSJN. Solo interrumpe si la tarea entrante es más corta que lo que le falta por terminar a la tarea actual.
  • RR (Round Robin): Reparto por "Quantum" (q). Cada tarea recibe un trozo de tiempo. Es el más justo para usuarios interactivos, pero si q es muy pequeño, el costo de conmutar (overhead) arruina el rendimiento.

3. Gestión del Tiempo y Hardware Subyacente

Eventos del Sistema

  • Interrupción (Hardware) Es un evento asíncrono generado por componentes externos (teclado, disco, reloj). No avisan cuándo ocurrirán. Detienen la CPU inmediatamente, salvan el contexto y saltan a la tabla de vectores de interrupción (INTR).
  • Excepción (Software) Es un evento síncrono y anormal originado por el código mismo en tiempo de ejecución (ej. División por cero, Fallo de segmento de memoria).

Reloj y Temporizador

El sistema confía en un cristal de cuarzo físico (Hardware) que oscila a una frecuencia fija (Ej: 1GHz = 109 pulsos/segundo).

  • Registro Reloj: Se incrementa con cada pulso del cuarzo. Mantiene la Hora Universal (UTC / Epoch).
  • Temporizador (Watchdog / RTEMPO): Es un registro que cuenta en reversa. El SO lo inicializa en un valor llamado RTMAX. Se decrementa con cada pulso.
  • El evento clave: Cuando RTEMPO llega a 0, dispara por hardware la Interrupción del Reloj (IRQ 0).

La Rutina de Interrupción y el TICK

El tiempo transcurrido entre dos IRQ 0 consecutivas se llama TICK (ej. 1ms o 10ms). Cada vez que esto ocurre, el procesador deja lo que estaba haciendo y ejecuta una rutina crítica de gestión. El tiempo gastado en esta rutina se conoce como Overhead.

¿Qué hace el SO en cada TICK?
  1. Actualizar la hora universal.
  2. Revisar la Timers List (lista de plazos) para despertar tareas (ej. sleep(5)).
  3. Actualizar la contabilidad de CPU de la tarea actual.
  4. Decrementar el Quantum. Si llega a 0, forzar la salida de la tarea (Scheduling).
Corrección de Hora

La ejecución de la rutina de interrupción toma tiempo. Durante ese tiempo, RTEMPO pasa a valores negativos. Para que el reloj no se atrase severamente, se ajusta con:

Hora = Hora + TICK - |RTempo|

Aun así, pequeñas instrucciones de HW no medibles atrasan el reloj. Por eso se usan protocolos como NTP (Network Time Protocol) que ajustan el reloj de a poco (Drifting) en vez de hacerlo saltar de golpe.

Parte 2: Preguntas Teóricas (Tipo Certamen)

Preguntas extraídas directamente de pautas de certámenes anteriores. Haz clic en cada pregunta para revelar la respuesta.

El modelo de planificación determina la filosofía y forma de uso del procesador (cuándo se pierde, durante cuánto tiempo una tarea lo tiene, si hay reciclaje o quantum). El algoritmo de scheduling es la matemática que elige cuál es la tarea siguiente cuando la tarea en ejecución actual pierde el procesador (porque terminó, hizo E/S, o se le acabó su quantum).

En Shortest Job Next (SJN), la prioridad siempre la tiene la tarea más corta. Si en un sistema existe un flujo constante y continuo de tareas cortas llegando a la cola "LISTA", las tareas largas siempre quedarán relegadas al final. Como consecuencia, las tareas largas jamás obtendrán la CPU y sufrirán de inanición indefinida. Para solucionar esto se creó el HPRN, que aumenta la prioridad según el tiempo de espera.

Aunque el SO corrige la hora sumando el tiempo que demoró la rutina de interrupción misma (usando el valor negativo con el que quedó el registro RTEMPO), existen instrucciones a muy bajo nivel (como rearmar/resetear el temporizador de vuelta a RTMAX y la instrucción return final que devuelve el control de la interrupción) cuyo tiempo de ejecución de Hardware no se puede medir ni sumar a la variable "Hora". En cada TICK se pierden unos pocos nanosegundos invisibles al SO, lo que a largo plazo (días/semanas) atrasa el reloj.

Si el reloj local del computador saltara abruptamente hacia el futuro (por ejemplo, de las 23:59 directamente a las 00:05), los tiempos intermedios dejarían de existir para el Sistema Operativo. Esto causaría que cualquier tarea crítica, servicio o script programado para ejecutarse en ese lapso exacto (ej. un respaldo (cronjob) programado a las 00:00) jamás se ejecute. Por eso, el protocolo NTP usa una técnica de "Drifting", acelerando o ralentizando los TICKS sutilmente para alcanzar la hora real gradualmente sin perder ningún segundo.

  • TICK Pequeño: Aumenta drásticamente la precisión del tiempo y hace el sistema más responsivo. Desventaja: Genera demasiadas interrupciones (IRQ0). El procesador gasta gran parte de su capacidad solo ejecutando las rutinas del SO en vez de procesar tareas de usuario. Sube el Overhead, baja el rendimiento útil.
  • TICK Grande: Minimiza el número de interrupciones. El procesador dedica casi toda su potencia al trabajo útil (Bajo overhead). Desventaja: Se pierde resolución y precisión temporal. Un usuario interactivo sentirá el sistema trabado o lento para responder a eventos inmediatos.

3.1 Comparativa de Rendimiento: SJN vs PSJN

Considere las siguientes 5 tareas (A, B, C, D, E) con sus respectivos tiempos de llegada y tiempos estimados de ejecución:

1. Modelo SJN (Batch - Sin Requisición)

La política dicta: "El que entra no sale hasta terminar". En t=0, A(5) y B(4) llegan juntas. Como B es más corta, se ejecuta primero y monopoliza la CPU hasta t=4. En t=4 analizamos la cola: están esperando A(5) y la recién llegada C(3). Entra C. Y así sucesivamente priorizando el tamaño.

B (4)
C (3)

A (5)
047101217

T. Respuesta (Tfin - Tllegada):

  • A: 17 - 0 = 17ms
  • B: 4 - 0 = 4ms
  • C: 7 - 3 = 4ms
  • D: 12 - 8 = 4ms
  • E: 10 - 5 = 5ms

T. Penalización (Tresp / Duración):

  • A: 17 / 5 = 3.4 (Alta penalidad)
  • B: 4 / 4 = 1.0 (Perfecto)
  • C: 4 / 3 ≈ 1.33
  • D: 4 / 2 = 2.0
  • E: 5 / 3 ≈ 1.66

2. Modelo PSJN (Con Requisición)

La política dicta: "Si llega una tarea nueva MÁS CORTA que la actual, expropia la CPU". En t=0 entra B(4). En t=3 llega C(3); en este instante, a B solo le queda 1ms, por lo que 1 < 3. B NO es interrumpida. Termina B en t=4 y entra C. ¡Ocurre lo mismo en toda la traza! Casualmente este set de datos no provoca interrupciones a medio camino, comportándose idéntico a SJN.

Ejercicio 2: Ayudantía 3 (Gestión de Quantum)

3.2 Gestión del Quantum y Costo de Conmutación (Overhead)

Considere un sistema de Tiempo Compartido donde las tareas consumen la totalidad de su Quantum. Existe un costo o penalidad de hardware/software cada vez que el SO realiza un cambio de contexto (Task Switching), representado por s = 1ms.

Pregunta: Si se configura un Quantum (q) = 100ms, evalúe el impacto del overhead (tiempo total de vida vs tiempo útil) para una tarea T1 = 20ms y otra tarea T2 = 500ms.

El "Overhead" es tiempo procesador que NO avanza el trabajo del usuario. Se calcula analizando cuántas conmutaciones sufre la tarea a lo largo de su vida.

Análisis de T1 (20ms)

  • Como 20ms < 100ms, la tarea termina en su primer quantum.
  • Genera 1 sola conmutación (para salir).
  • Tiempo Total Vida: 20ms + 1ms = 21ms
Pérdida (Overhead) = 1ms / 21ms ≈ 4.76%

Análisis de T2 (500ms)

  • Como consume 500ms, agotará el quantum de 100ms un total de 5 veces.
  • Genera 5 conmutaciones obligadas por el sistema.
  • Tiempo Total Vida: 500ms + (5 * 1ms) = 505ms
Pérdida (Overhead) = 5ms / 505ms ≈ 0.99%
Conclusión de Diseño: Si el Quantum q es muy pequeño, la relación s/q crece y el rendimiento se desploma por el exceso de conmutaciones. Si es muy grande, baja la pérdida (como en T2), pero aumenta el tiempo que deben esperar las tareas interactivas nuevas.
Ejercicio 3: Ayudantía 3 (Arquitectura)

3.3 Frecuencia y Registro Temporizador (RTMAX)

Un procesador de 64 bits dispone de un registro temporizador (RTEMPO) que funciona en complemento 2. Está asociado a un cristal de cuarzo de frecuencia F = 1 GHz y se decrementa en 1 en cada pulso de reloj. El sistema operativo quiere establecer un TICK exacto de 10 milisegundos (10ms).

Pregunta: ¿Con qué valor inicial (RTMAX) debe configurar el SO al registro RTEMPO para asegurar la IRQ 0 en el tiempo preciso?

Para resolver esto, debemos normalizar las unidades a la base matemática (segundos).
Un cuarzo de 1 GHz emite 109 pulsos por segundo. Queremos que nuestro contador de cuenta regresiva llegue a cero en 10ms, que es 10-2 segundos.

Fórmula: RTMAX = Frecuencia (Hz) × TICK deseado (s)

RTMAX = (109 Hz) × (10-2 s)

RTMAX = 107 = 10.000.000

Es decir, el SO cargará el número diez millones en el HW. El hardware restará uno a la velocidad de mil millones de veces por segundo, llegando a 0 y disparando la interrupción exactamente 10 milisegundos después.

Ejercicio 4: Ayudantía 3 (Cálculo de Overhead)

3.4 Porcentaje Global de Overhead del Sistema

Es vital medir qué porcentaje del tiempo la CPU gasta haciendo labores de administración (overhead) versus labores útiles de los usuarios. Consideremos la siguiente rutina del Sistema Operativo:

  • Cada TICK: Actualiza hora y timers. Demora Δt1 = 100 μs.
  • Cada K=5 TICKS: Actualiza prioridades. Demora Δt2 = 150 μs.
  • Cada N=10 TICKS: Quantum expira. Ocurre Scheduling y Context Switch. Demora Δt3 = 500 μs.

Pregunta: Si el TICK base es de 10ms, ¿Cuál es el porcentaje p de la CPU que se desperdicia en Overhead en un ciclo completo de 1 Quantum (N)?

El ciclo de estudio es sobre 1 Quantum (N = 10 TICKS). Durante estos 10 TICKS, los eventos de administración ocurren con diferente frecuencia. Debemos sumar todos esos tiempos perdidos y dividirlos por el tiempo real del reloj.

Fórmula del Overhead Total (OH)

OH = (N × Δt1) + ((N/K) × Δt2) + (1 × Δt3)

Explicación: N=10 veces ocurre Δt1. N/K = 10/5 = 2 veces ocurre Δt2. 1 sola vez ocurre el Δt3 al final del quantum.

OH = (10 × 100μs) + (2 × 150μs) + (500μs)

OH = 1000μs + 300μs + 500μs = 1800 μs (Tiempo Perdido)

Cálculo del Porcentaje (p)

Tiempo Total del Ciclo (TT) = N × TICK = 10 × 10ms = 100ms = 100.000 μs

p = OH / TT

p = 1800 / 100000 = 0.018 = 1.8%
