Sistemas Operativos

Ayudantía 1: Planificación de Tareas y Multiprogramación

Materia y Conceptos

  • Un programa es la expresión de un algoritmo con ayuda de un lenguaje de programación.

  • Una tarea es la unión de un programa con los datos a los cuales se aplica.

  • Un proceso es la ejecución de una tarea. Se caracteriza por la tarea y su contexto de ejecución, es decir, el valor del Program Counter (registro PC), los valores de los otros registros del procesador, etc...

Mono-Programación

Hay solamente un proceso al instante t en memoria. Cuando se somete una tarea, el sistema la carga y ejecuta el proceso correspondiente hasta el final, pasando luego a la siguiente tarea.

Multiprogramación

Se puede tener más de un proceso al instante t en memoria. Los procesos que están listos se ejecutan consecutivamente (conmutación de procesos). Puede usarse en Batch Processing o en Tiempo Compartido.

Tipos de Conmutación:

  • En Batch Processing con E/S, solo hay conmutación si el proceso activo debe efectuar una E/S.
  • En Tiempo Compartido, hay conmutación si el proceso activo espera un evento o si se termina su quantum de tiempo.

Nota: El Quantum es elegido por el sistema operativo durante el Boot (del orden de 10 a 100ms).

Una unidad de intercambio (DMA) sirve para efectuar transferencias entre la memoria principal y un periférico sin usar el procesador. Recibe una orden, transfiere los datos y avisa mediante una interrupción. Durante la transferencia, el procesador puede hacer otros cálculos.

2. Tasa de Ocupación del Procesador

Datos del problema:

  • T1: dura 200ms, única E/S al t=110ms. (Llega t=0)
  • T2: dura 50ms, única E/S al t=5ms. (Llega t=0)
  • T3: dura 120ms sin E/S. (Llega t=140ms)
  • Cada E/S dura 10ms. Quantum q = 100ms para Time Sharing.

Cálculos:

  • T1: Empieza t=0. Hace E/S en t=110 (termina su E/S en 120).
  • T2: Retoma la CPU en t=110. Hace E/S en t=115 (termina su E/S en 125).
  • Entre 115 y 120, ambas están bloqueadas en E/S → IDLE (5ms).
  • T1 retoma en 120, termina en 210.
  • T2 retoma en 210, termina en 255.

Tiempo Total (T) = 375ms

Tiempo Ocupado = 375 - 5 (IDLE) = 370ms

Tp = 370 / 375 = 0.9866 (98.66%)

Pregunta 2.3

Mono-programación (1 Tarea)

Una única tarea (T1) se somete en t=0. Represente la Gantt. Calcule el tiempo total de ejecución T, Tp (tasa CPU) y Tu (tasa disco).

Ciclo: L (8) + C (7) + E (5) = 20ms. Como N=4, el tiempo total es 4 * 20ms = 80ms.

Cálculos:

T (Total) = 80ms

Tiempo CPU ocupada = 4 * C = 4 * 7ms = 28ms.

Tiempo Disco ocupado = 4 * (L + E) = 4 * (8 + 5) = 52ms.

Tp = 28 / 80 = 0.35 (35%)

Tu = 52 / 80 = 0.65 (65%)

Pregunta 2.4

Mono-programación (2 Tareas)

¿Qué se puede deducir de T, Tp y Tu cuando las tareas T1 y T2 se ejecutan en mono-programación?

En mono-programación, T2 debe esperar completamente a que T1 termine sus 4 ciclos antes de poder comenzar.

  • Tiempo Total (T): Se duplica exactamente. T = 80ms + 80ms = 160ms.
  • Tasas de ocupación (Tp y Tu): Se mantienen exactamente iguales (35% y 65% respectivamente).
  • Conclusión: El hardware está profundamente subutilizado. Mientras la CPU calcula, el disco no hace nada; mientras el disco lee/escribe, la CPU está ociosa. No hay solapamiento de operaciones.
Pregunta 2.5

Batch Multiprogramación (T1 y T2)

Gantt para T1 y T2 en Multiprogramación Batch (C < L y C > E). Calcule T, Tp, Tu y compare.

Aquí el disco y la CPU pueden trabajar al mismo tiempo. T2 puede usar la CPU mientras T1 usa el disco, y viceversa.

Tiempo 0-8 8-15 15-20 20-23 23-30 30-35
CPU IDLE T1 (C:7) T1 (IDLE) T2 (C:3) T2 (C:4) ...
Disco T1 (L:8) T2 (L:7) T2 (L:1) / T1(E:4) T1 (E:1) T1 (L:7) ...

Al traslapar (pipeline) las operaciones, el tiempo total se reduce drásticamente en comparación a los 160ms de mono-programación.

T (aprox) ≈ 88ms

Tp = (28 + 28) / 88 = 56 / 88 = 63.6%

Tu = (52 + 52) / 88 = 104 / 88 = 100% (Saturado)

Comparación: Tp saltó de 35% a 63%, y Tu de 65% a 100%. El sistema es mucho más eficiente.

3. Tiempo de Respuesta en Modo Batch

Parámetros del Sistema:

  • n usuarios ejecutando comandos de edición.
  • Cada comando consume c segundos de procesador.
  • r es el tiempo de respuesta (desde solicitud hasta respuesta).
  • t es el tiempo de reflexión/tipeo antes de someter nuevo comando.

*Nota: Asumiremos los valores de t=2s y c=0.1s para que el ejercicio tenga sentido físico.

Pregunta 3.1

Cálculo en Régimen Permanente

Todos los usuarios comienzan a trabajar al mismo tiempo. Exprese el tiempo de respuesta en función de los otros parámetros en régimen permanente. Calcule r para n=10, t=2s y c=0.1s y luego para n=30, t=2s y c=0.1s.

En régimen permanente, el ciclo total de un usuario es r + t. El sistema puede procesar 1/c trabajos por segundo, y hay n usuarios enviando trabajos.

  • Caso No Saturado (n*c ≤ t): El sistema es capaz de procesar a todos antes de que vuelvan a pensar. No hay cola.
    Fórmula: r = c
  • Caso Saturado (n*c > t): El procesador está al 100%. El tiempo de ciclo de un usuario se vuelve igual al tiempo necesario para procesar a todos (n*c). Por tanto: r + t = n*c.
    Fórmula: r = n*c - t

Cálculo 1: n=10, t=2s, c=0.1s

Evaluamos saturación: n * c = 10 * 0.1s = 1s.

Como 1s < 2s (t), el sistema NO está saturado.

r = c = 0.1s

Cálculo 2: n=30, t=2s, c=0.1s

Evaluamos saturación: n * c = 30 * 0.1s = 3s.

Como 3s > 2s (t), el sistema SÍ está saturado.

r = n*c - t = 3s - 2s = 1s

Pregunta 3.2

Trabajo Interactivo en Batch

¿Es razonable pensar en un trabajo interactivo en modo Batch?

No, no es razonable.

En modo Batch (Procesamiento por Lotes), una tarea se ejecuta ininterrumpidamente hasta terminar. Si un usuario envía un comando largo o un trabajo pesado (por ejemplo, un cálculo matemático de 5 minutos), el procesador no lo soltará.

Todos los demás usuarios "interactivos" (como los que editan texto) quedarán congelados durante esos 5 minutos sin poder tipear una sola letra. Un sistema interactivo exige Tiempos de Respuesta (r) bajos y predecibles, lo cual Batch no puede garantizar si se mezclan con trabajos largos.

4. Tiempo de Respuesta en Tiempo Compartido

Nuevos Parámetros:

  • No hay overhead por conmutación. T = t = 0 (No hay tiempo de reflexión, los usuarios envían procesos continuamente).
  • n usuarios realizando edición (consumen c).
  • N usuarios efectuando compilaciones (consumen C).
  • c y C son múltiplos del quantum q.
Pregunta 4.1

Cálculo en Régimen Permanente

Expresar el tiempo de respuesta r (para edición) y R (para compilación), en función de los otros parámetros. Calcule r y R para n=10, c=0.1s, N=5 y C=3s.

Como t=0, siempre hay (n + N) tareas en la cola de listos en todo momento.

El algoritmo Round Robin (Tiempo Compartido) le da un "turno" a cada una de las (n+N) tareas. Por lo tanto, para acumular 1 segundo de CPU, una tarea tiene que esperar a que pasen todos los demás.

Fórmula general (con t=0):
Tiempo de Respuesta = (Tiempo requerido de CPU) × (Número total de usuarios compitiendo)
r = c * (n + N)
R = C * (n + N)

Cálculo de Tiempos

Usuarios totales en competencia = n + N = 10 + 5 = 15.

Editores (r):

r = c * (n + N) = 0.1s * 15 = 1.5s

Compiladores (R):

R = C * (n + N) = 3s * 15 = 45s

Pregunta 4.2

Conclusión

Concluir sobre el Tiempo Compartido comparado al Batch Processing.

El Tiempo Compartido (Time Sharing) es infinitamente superior para cargas de trabajo mixtas e interactivas.

  • Equidad y Evita la Inanición: En Batch, un trabajo de compilación (3s) bloquearía a los editores durante 3 segundos enteros. Si hubieran 5 compiladores seguidos, el editor esperaría 15 segundos antes de reaccionar.
  • Respuesta Interactiva: En Tiempo Compartido, el sistema divide el tiempo en quantums. Aunque hay procesos pesados compilando de fondo, el editor recibe porciones de CPU constantemente, logrando un tiempo de respuesta de 1.5s, haciendo que el sistema siga sintiéndose "vivo" y respondiente para el usuario.
  • Degradación Justa: Las tareas cortas terminan rápido (1.5s), mientras que las pesadas toman su tiempo en el fondo (45s), pero ninguna bloquea totalmente a la otra.
