Parte 1 · Objetivos de la Memoria Virtual

1.1 ¿Por qué Memoria Virtual?

En la gestión de memoria lineal, el proceso completo debe cargarse en RAM para ejecutar. Esto genera tres problemas principales:

Espacios monolíticos

No hay separación código/datos/pila. Sin estructuración → difícil de proteger.

Saturación rápida

Cargar todo el proceso tiene un costo enorme en memoria cuando el espacio de trabajo es dinámico y grande.

Swapping costoso

El intercambio completo proceso↔disco es muy lento. Compactar también tiene un alto costo de tiempo.

Solución: Memoria Virtual

Usar direcciones virtuales relativas al espacio de trabajo de cada proceso, cargar sólo las partes necesarias, y verificar mediante hardware (MMU) que el proceso no salga de su espacio virtual.

1.2 Objetivos Principales

Transparencia

El proceso no sabe en qué dirección física está cargado. La MMU traduce automáticamente cada acceso virtual → físico. El programador trabaja sólo con el espacio virtual.

Protección

Cada proceso tiene su propio espacio virtual. La MMU verifica que no acceda fuera de él. Si lo hace, genera un Segmentation Fault o Page Fault.

Compartición

Varios procesos pueden compartir segmentos (ej. biblioteca libc) cargados una sola vez en memoria física, apuntados desde tablas locales de distintos procesos.

Eficiencia (On Demand)

Se carga una página sólo cuando hay un acceso a ella. El principio de localidad garantiza que normalmente basta cargar pocas páginas para ejecutar bien.

1.3 La MMU — Memory Management Unit

La MMU es el componente de hardware del procesador que intercepta cada acceso a memoria y realiza:

  1. Traducción: Dirección virtual → Dirección física (usando tabla de páginas/segmentos).
  2. Verificación de permisos: ¿Tiene el proceso derecho R/W/X sobre este segmento/página?
  3. Generación de excepciones: Page Fault, Segment Fault, Protection Fault.
  4. Actualización de bits: Bit U (Used) y bit D (Dirty) en la tabla.

1.4 Principio de Localidad

El uso de las páginas NO es uniforme: sigue patrones que justifican el On Demand Paging.

Localidad Temporal

Un elemento recientemente usado tiene alta probabilidad de ser accedido nuevamente muy pronto. Ejemplos: variables de loop, contadores, funciones recursivas.

Localidad Espacial

Los elementos adyacentes al elemento accedido también tienen alta probabilidad de ser usados. Ejemplos: arreglos, structs, código secuencial, strings.

⇒ Gracias a la localidad, mantener en RAM sólo las páginas del "Working Set" activo es suficiente para una buena performance. No es necesario cargar todo el proceso.

Parte 2 · Paginación y Tabla de Páginas

2.1 Conceptos Fundamentales

Página

Unidad del espacio virtual del proceso. Tamaño fijo (ej. 4KB, 32KB). Índice: número de página virtual p.

Frame

Unidad de la memoria física (RAM). Mismo tamaño que la página. Índice: número de frame f.

Offset

Desplazamiento d dentro de la página/frame. Se mantiene igual en la traducción virtual→física.

Dirección Virtual = (p, d)  →  @Física = frame[p] × TamañoPágina + d

2.3 Page Fault — Ciclo Completo

1. Acceso: La CPU genera una dirección virtual (p, d). La MMU consulta la tabla de páginas.

2. Bit V=0: La MMU detecta que la página no está en RAM → lanza excepción Page Fault (trap al SO).

3. SO interviene: Busca la página en disco (swap). Si hay un frame libre, lo usa. Si no hay frames libres → ejecuta el algoritmo de reemplazo para elegir víctima.

4. Swap Out (víctima): Si la página víctima tiene D=1 (Dirty), el SO la escribe en disco antes de liberar el frame.

5. Carga: El SO carga la página requerida en el frame liberado. Actualiza tabla: V=1, frame=f.

6. Reanuda: El SO retoma la instrucción que falló. Ahora la MMU puede completar la traducción correctamente.

2.4 Cálculo del Número de Páginas Virtuales

Para saber cuántas páginas necesita un proceso, se divide cada sección por el tamaño de página (redondeando al entero superior):

N° páginas = ⌈ Tamaño_sección / Tamaño_página ⌉

Ejemplo — Proceso P0, páginas de 32 KB = 32 768 bytes:

• Código: 124 800 B ÷ 32 768 = 3,81 → 4 páginas

• Datos: 200 000 × 4 B = 800 000 B ÷ 32 768 = 24,41 → 25 páginas

• Stack/Heap: 64 KB ÷ 32 KB = 2,00 → 2 páginas

Total: 4 + 25 + 2 = 31 páginas virtuales

Parte 3 · Algoritmos de Reemplazo de Páginas

Cuando la memoria está saturada (todos los frames ocupados) y ocurre un Page Fault, el SO debe elegir qué página descargar (página víctima). El objetivo: minimizar los Page Faults futuros. Los algoritmos que veremos son aproximaciones del óptimo teórico (que requiere conocer el futuro).

FIFO First In, First Out

▼

La página que lleva más tiempo en memoria es la primera en salir.

Funcionamiento

  • El SO mantiene una cola FIFO (lista ordenada por hora de carga).
  • Al necesitar reemplazar, la víctima es la cabeza de la cola (la más antigua).
  • La nueva página entra al final de la cola.
  • Los frames se reciclan en orden circular (Round Robin).

Evaluación

OK Muy simple de implementar, sin hardware especial.
OK Bajo costo computacional.
NO No considera si la página sigue siendo útil.
NO Sufre la Anomalía de Belady: más frames puede producir más Page Faults.

Anomalía de Belady

Con la secuencia 1,2,3,4,1,2,5,1,2,3,4,5 y 3 frames → 9 Page Faults. Con 4 frames → ¡10 Page Faults! Esto no pasa en LRU ni en OPT.

LRU Least Recently Used

▼

La página que hace más tiempo que no se usa es la víctima.

Funcionamiento

  • Mantiene una lista ordenada por último acceso.
  • Cada acceso mueve la página al frente de la lista.
  • La víctima es la que está al final (menor uso reciente).
  • Requiere soporte hardware (TLB con timestamps) para ser eficiente.

Evaluación

OK Toma en cuenta la utilidad reciente de las páginas.
OK No sufre la Anomalía de Belady.
OK Muy buena aproximación al algoritmo óptimo.
NO Requiere hardware especial (TLB) → costoso.
NO En práctica se usa aproximación FINUFO.

NFU Not Frequently Used (Second Chance)

▼

Aproximación de LRU usando el bit U. La página con U=0 es candidata.

Funcionamiento (Second Chance)

  • Usa los bits V (Valid) y U (Used) de cada página.
  • Recorre la cola en orden FIFO.
  • Si la cabeza tiene U=0 → es la víctima.
  • Si la cabeza tiene U=1 → le da una segunda oportunidad: pone U=0 y la mueve al final de la cola.
  • Si todas tienen U=1 → hace una vuelta completa y resetea todos los U. La primera en la cola es víctima.

Evaluación

OK Compromiso entre FIFO y LRU.
OK Sin hardware especial (sólo bits U y V).
OK Costo sólo en Page Fault y carga inicial.
NO Mayor costo que FIFO puro.
NO No distingue el orden de acceso dentro de las páginas con U=1.

LFU Least Frequently Used

▼

La página con menor frecuencia de uso acumulado es la víctima.

Funcionamiento

  • Cada página tiene un contador de accesos.
  • La víctima es la página con el contador más bajo.
  • En empate, se puede usar FIFO como desempate.
  • Implementado con un contador de 8 bits que se desplaza (shift right) periódicamente.

Evaluación

OK Considera la frecuencia de uso a largo plazo.
NO Una página muy usada en el pasado pero no en el presente puede "acumular crédito" falso.
NO Necesita más memoria para los contadores.

WS Working Set

▼

Mantener en RAM el conjunto activo de páginas de cada proceso.

El Working Set W(t, Δt) es el conjunto de páginas accedidas por el proceso en los últimos Δt instantes de tiempo.

W(t, Δt) = {páginas accedidas en el intervalo (t−Δt, t]}

Thrashing

Si los frames asignados a un proceso son menores que el tamaño de su Working Set, el proceso genera Page Faults constantemente. El SO dedica más tiempo gestionando fallos que ejecutando procesos → thrashing (colapso).

PFF — Page Fault Frequency

Controla dinámicamente los frames asignados según la tasa de fallos:

  • Si tasa > umbral_alto → asignar más frames al proceso.
  • Si tasa < umbral_bajo → liberar frames del proceso.
  • Objetivos: minimizar Page Faults sin desperdiciar memoria.

Ej. 1

Cálculo de páginas y Page Faults en un loop

Paginación

Procesador 32 bits. RAM 4GB. Páginas de 32 KB. Un int ocupa 4 bytes.

El proceso P0 tiene: Código = 124 800 B, Stack = 64 KB, Datos = int tab[200000].

El proceso ejecuta: for(int i=0; i<200000; i+=40000) tab[i]=i;

¿Cuántos Page Faults provoca este loop?

Ver Solución ▼

Paso 1: ¿Dónde queda cada elemento del arreglo?

Tamaño total de datos = 200 000 × 4 B = 800 000 B.

Las páginas de datos empiezan después de las 4 páginas de código. La página virtual de datos #0 empieza en el byte 0 del segmento de datos.

Paso 2: Dirección de cada acceso del loop

i=0: tab[0] → byte 0 → página datos #0 (0 ÷ 32768 = 0)

i=40000: tab[40000] → byte 160000 → página datos #4 (160000 ÷ 32768 = 4,88 → pág 4)

i=80000: tab[80000] → byte 320000 → página datos #9 (320000 ÷ 32768 = 9,76 → pág 9)

i=120000: tab[120000] → byte 480000 → página datos #14 (480000 ÷ 32768 = 14,6 → pág 14)

i=160000: tab[160000] → byte 640000 → página datos #19 (640000 ÷ 32768 = 19,5 → pág 19)

Resultado: 5 Page Faults

El loop accede a 5 páginas virtuales distintas (páginas 0, 4, 9, 14, 19 relativas al inicio de datos). Cada primera vez en una página genera un Page Fault.

Parte 6 · Comparación de Algoritmos

Algoritmo Política Hardware Costo Anomalía Belady Calidad
FIFO Más antiguo en memoria Ninguno Muy bajo Sí Baja
LRU Menos recientemente usado TLB + timestamps Alto No Muy alta
NFU Primera con U=0 (2nd chance) Bits U y V Medio No Buena
LFU Menor frecuencia acumulada Contadores Medio No Buena
OPT No usada por más tiempo Requiere futuro Imposible No Óptima

En la práctica: Linux usa una variante de NFU/LRU aproximado con bits de hardware (bit A en x86). Windows usa Working Set + NFU. Los algoritmos puros de LRU son demasiado costosos para implementar a nivel de hardware de paginación, por eso se usan aproximaciones.
