1 — Sistema de Computación y el Kernel

¿Qué debe hacer un sistema operativo?

El SO controla la ejecución correcta de múltiples tareas y el funcionamiento del hardware: asigna memoria, crea/inicializa/termina tareas, gestiona recursos físicos y lógicos, y reacciona a eventos múltiples — system calls, fallas de tareas, interrupciones y excepciones.

Tres niveles de gestión

Corto plazo

Reacciones a eventos, inicialización de tareas, protección. Ocurre constantemente, a nivel de microsegundos.

Mediano plazo

Gestión de memoria/discos, reducción del Idle Wait, scheduling, concurrencia entre tareas.

Largo plazo

Gestión global de recursos: creación/inicialización de tareas, asignación y distribución de recursos.

El Kernel

Es la parte del sistema que efectúa las operaciones básicas de gestión del hardware y de las tareas: corrección de instrucciones, accesos a memoria, control de periféricos y gestión del tiempo. El hardware se encarga de gran parte de esto por sí solo — el kernel interviene excepcionalmente, ante fallas detectadas por el hardware.

El hardware (controladores de periféricos, MMU, reloj) ejecuta solo la mecánica de bajo nivel: mover bits, generar una señal cuando termina una E/S, decrementar un contador. No decide nada sobre qué proceso debe correr ni cómo reaccionar a un error.

Lo hace solo el hardware

  • Ejecutar la instrucción actual
  • Generar la señal de interrupción
  • Avanzar el reloj / temporizador

Requiere al kernel

  • Decidir qué hacer cuando ocurre la interrupción
  • Elegir el próximo proceso (scheduling)
  • Tratar fallas y recursos faltantes
  • Corto plazo: al terminar una E/S de disco, el kernel atiende la interrupción y despierta el proceso bloqueado (milisegundos).
  • Mediano plazo: el scheduler decide hacer swap-out de un proceso SLEEPING para liberar memoria física para otro READY (decenas de milisegundos a segundos).
  • Largo plazo: al hacer login, el sistema crea el proceso de la shell, le asigna su PID, memoria inicial y recursos (una vez, al inicio de la sesión).

2 — Interfaces con el Hardware

Dos mecanismos de comunicación

① Puertos de Entrada/Salida

Cada periférico y circuito tiene uno o más puertos de E/S. El procesador tiene instrucciones especiales para leer y escribir datos a través de estos puertos directamente.

② Interrupciones

Cuando un periférico termina su E/S, ocurre un error, o el reloj marca un tick, se genera una interrupción que el procesador debe atender.

Camino de datos típico (Intel/PC)

CPU ↔ CACHE
RAM ↔ Memory Controller
↓ CPU BUS ↓
PCI Express Controller
↓ CPU BUS ↓
Hard Disk Controller → Data Store
Controladores → Periféricos (teclado, impresora, red...)

Si el procesador tuviera que leer repetidamente un puerto de E/S para saber si el periférico terminó (polling), estaría desperdiciando ciclos de CPU que podría usar otro proceso. Con interrupciones, el procesador ejecuta otro proceso mientras espera, y solo se detiene cuando el periférico realmente tiene algo que entregar — esto reduce el Idle Wait y mejora el uso global de la CPU.

3 — Interrupciones y Excepciones

Interrupción

Evento hardware esperado por el sistema operativo: fin de una E/S, el reloj, etc. Es parte del funcionamiento normal.

Excepción

Evento hardware inesperado, resultante de un error: división por cero, violación de derechos de acceso, falla de hardware.

Dos tipos de excepciones

Asincrónicas

No están ligadas a la instrucción que se ejecutaba: error de un periférico (sector dañado), falla de una placa de memoria/procesador, fallas de alimentación eléctrica.

Sincrónicas

Provocadas directamente por la instrucción en curso: violación de protecciones (segmentation violation, FPU error), zero divide, falta de datos en memoria.

EventoCategoría

Segmentation faultExcepción sincrónica
Fin de lectura de discoInterrupción
Falla de alimentación eléctricaExcepción asincrónica

Una excepción asincrónica (ej. falla de una placa de memoria, sector de disco dañado) ocurre por una causa externa al flujo de instrucciones que el procesador está ejecutando en ese instante — el procesador puede estar en cualquier punto del programa cuando el hardware avisa del problema. En cambio, una excepción sincrónica (zero divide) es consecuencia directa de la instrucción que se acaba de ejecutar, por lo que el PC apunta exactamente a la instrucción culpable.

4 — Interrupt Masking y Operaciones Atómicas

Operación atómica

Una operación que se hace completamente o no se hace — nunca queda a medio camino. Una vez que empieza, no se puede interrumpir.

Para garantizar atomicidad, el sistema usa el IRQ Controller para prohibir momentáneamente una o más interrupciones (MASKING) y luego restablecerlas (UNMASKING). Una interrupción enmascarada no se pierde: queda en una lista de interrupciones pendientes hasta que se desenmascara.

Depende de qué línea IRQ está enmascarada. El masking es por línea: si solo se enmascaró la línea del reloj, las interrupciones de disco (otra línea) se atienden normalmente y no entran en la lista de pendientes.

Si en cambio se enmascarara también la línea del disco, las 3 interrupciones de disco entrarían en la lista pending del IRQ Controller. Al desenmascarar, el controlador entrega la(s) interrupción(es) pendiente(s) al procesador — el SO debe estar preparado para procesarlas, aunque típicamente solo se necesita atender una vez por línea (las señales repetidas del mismo dispositivo suelen coalescer).

Porque algunas operaciones del kernel —como modificar una estructura compartida (lista de procesos READY, tabla de páginas)— deben verse como atómicas. Si una segunda interrupción llegara a mitad de esa modificación, podría dejar la estructura en un estado inconsistente (ej. un puntero a medio actualizar). El masking convierte ese tramo crítico en una sección que no puede ser interrumpida, garantizando la integridad de las estructuras del kernel.

5 — Contexto y Conmutación de Procesos

Contexto de un proceso (instante t)

El valor de todos los registros del procesador, su espacio de trabajo en memoria (Address Space) y sus pilas de ejecución (proceso + sistema).

Task Switching (Conmutación de tarea)

1. Salvar contexto

Del proceso actual en ejecución, en la pila sistema (no la pila usuario)

2. Elegir siguiente

El scheduler (schedule()) elige la próxima tarea READY

3. Restaurar contexto

Del proceso elegido — registros, stack, address space

¿Por qué Linux no usa las instrucciones hardware especiales para esto?

Existen instrucciones de procesador para salvar contexto, pero son lentas y no necesariamente salvan todos los registros. Linux usa una solución software: los registros se empujan a la pila de ejecución sistema del proceso con instrucciones push normales (macro SAVE_ALL), lo cual es más rápido y completamente controlado por el kernel.

cr3 es el registro de control que apunta a la tabla de páginas del proceso actual (task[j]→mm_struct en Linux). Al conmutar de la tarea i a la tarea j, el kernel actualiza cr3 para que apunte al espacio de direcciones de j — esto es lo que hace que las mismas direcciones virtuales ahora resuelvan a páginas físicas distintas, dando a cada proceso su propio espacio de memoria de forma transparente.

  • clock_intr(): el reloj interrumpe y el quantum del proceso actual terminó → se llama a schedule().
  • Entra a E/S: el proceso se bloquea esperando una operación → no puede seguir corriendo, se debe elegir otro.
  • Otro caso (Other Case): cualquier otro evento que requiera reevaluar qué proceso debe correr (ej. requisición explícita, señal).

Si el quantum no terminó y el proceso no está bloqueado, simplemente se retorna sin conmutar (rama "No" del diagrama).

6 — Modos de Control: System vs User

Los procesadores modernos (≥ i386) tienen dos modos de funcionamiento. El proceso no cambia de un modo a otro — solo cambia el nivel de privilegio con el que el procesador ejecuta sus instrucciones.

Capacidad System / Real Mode User / Protected Mode
Manipular tablas de páginasSíNo
Modificar registros del procesador (timers...)SíNo
Modificar tabla de interrupcionesSíNo
Ver direcciones reales de memoriaSíNo (solo virtuales)
¿Quién corre en este modo?El sistema operativo (kernel)Un proceso, salvo durante una syscall

Transiciones entre modos

USER → SYSTEM

Por TRAPS (sincrónicos) — el proceso mismo invoca una syscall — o por interrupciones (asincrónicas) — el hardware avisa de algo externo. Ambas pasan por la misma tabla de interrupciones.

SYSTEM → USER

Solo por creación de un proceso nuevo o por retorno (IRET) tras procesar la interrupción/TRAP — nunca de forma espontánea desde el kernel.

Esos bucles representan la ejecución normal, sin cambio de modo: mientras no ocurra un TRAP ni una interrupción, el procesador sigue ejecutando instrucciones del mismo tipo permitido en ese modo. En SYSTEM MODE el conjunto permitido es todas las instrucciones (privilegiadas y no privilegiadas); en USER MODE el hardware bloquea cualquier instrucción privilegiada — si el proceso intenta una, eso mismo genera un TRAP de protección que lo saca del bucle y lo lleva a SYSTEM MODE (normalmente para terminarlo).

7 — TRAPS y System Calls

TRAP

Evento sincrónico provocado por la ejecución de una instrucción. Dos causas posibles: (1) errores (overflow, zero divide, violación de protección) o (2) una llamada de sistema — el mecanismo que ofrece todos los servicios del SO a los procesos.

Una system call es una interrupción software: usa la misma tabla de interrupciones que el hardware. El hardware detecta la llamada, pasa el procesador a modo SYSTEM, ejecuta la rutina del kernel, y vuelve a modo USER con una instrucción de retorno (IRET, SYSEXIT o SYSRET según la arquitectura).

Mismo "Hello World", dos arquitecturas

32 bits — INT 0x80 (eax/ebx/ecx/edx)

; write syscall = 4, exit syscall = 1 movl $4, %eax ; código de servicio: write movl $1, %ebx ; arg1: fd = stdout movl $msg, %ecx ; arg2: buffer movl $12, %edx ; arg3: largo int $0x80 ; trap al kernel movl $1, %eax ; código de servicio: _exit movl $0, %ebx ; arg1: código de error int $0x80

64 bits — SYSCALL (rax/rdi/rsi/rdx)

; write syscall = 1, exit syscall = 60 movq $1, %rax ; código de servicio: write movq $1, %rdi ; arg1: fd = stdout movq $msg, %rsi ; arg2: buffer movq $12, %rdx ; arg3: largo syscall ; trap al kernel movq $60, %rax ; código de servicio: _exit movq $0, %rdi ; arg1: código de error syscall

Nota: en C nunca se escribe esto a mano — la libc (wrapper) traduce write() / _exit() a la secuencia correcta de registros + instrucción de trap para la arquitectura donde corre el binario.

TRAP por error

En algunos casos se repite la misma instrucción que falló, después de que el kernel resolvió el problema (ej. memoria virtual: el kernel trae la página faltante y reintenta el acceso). Si el kernel no puede resolverlo, termina el proceso.

TRAP por system call

Se continúa con la instrucción siguiente a la llamada (int $0x80 / syscall), normalmente recibiendo el resultado del servicio en el registro de retorno (ej. %eax/%rax).

Se mantiene

  • La lógica: registro = código de servicio, otros registros = argumentos
  • El mismo orden de argumentos (fd, buffer, largo)
  • Es una interrupción software, vía la tabla de interrupciones

Cambia

  • Instrucción de trap: int $0x80 → syscall
  • Registros usados: eax/ebx/ecx/edx → rax/rdi/rsi/rdx
  • Números de syscall distintos: write es 4 en 32 bits, pero 1 en 64 bits

Lo que las une: las tres pasan por la misma tabla de interrupciones del procesador y obligan el mismo ritual — masking, salvar contexto, ejecutar la rutina del SO, restaurar contexto, unmask, retorno (IRET/SYSEXIT/SYSRET). La diferencia está en quién dispara el evento y qué hace el SO una vez dentro.
